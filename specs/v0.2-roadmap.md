## REGLES ABSOLUES ‚Äî lire avant de coder

1. **Ne pas refactorer le code existant.** La v0.1 fonctionne. Ajouter du code, pas en reecrire. Si tu touches a `guard.ts`, tu AJOUTES dans `runGuard()` et `formatGuardResult()` ‚Äî tu ne changes pas ce qui existe.

2. **Ne pas creer de nouvelles dependances npm.** Tout se fait avec `ts-morph` (deja installe), `child_process` (Node.js natif), et les utilitaires existants dans `src/utils/`.

3. **Ne pas changer le format de l'index** (`index.json`). Le `FileNode` dans `base-parser.ts` ne change PAS. Les nouvelles donnees (historique git, hotspots, patterns) sont calculees a la volee, pas stockees dans l'index.

4. **Tester sur le vrai projet Cathodix** (`~/Desktop/dev/cathodix/`) apres chaque outil implemente. Pas juste des fixtures ‚Äî le vrai projet.

5. **Build apres chaque fichier modifie.** `npm run build` doit passer a chaque etape. Ne pas accumuler des modifications sans builder.

6. **L'utilisateur (Skhaall) ne code pas.** Il utilise Claude Code pour tout. Les messages de sortie des outils doivent etre comprehensibles par quelqu'un qui ne lit pas le code ‚Äî pas de jargon AST, pas de "CatchClause node", mais "catch vide ligne 42".

7. **Suivre l'ordre d'implementation** de cette spec (section "Ordre d'implementation recommande"). Ne pas sauter d'etape, ne pas commencer par le plus "interessant".

8. **Les hooks existants dans `~/.claude/settings.json` appellent `guard` et `check` automatiquement** a chaque Edit/Write. Donc les enrichissements de guard et check seront **immediatement actifs** sur tous les projets sans configuration supplementaire.

9. **Lire `specs/silent-catch.md`** avant d'implementer `silent_catch`. La spec detaillee est la-bas, pas ici.

## Pourquoi cette v0.2

### Le probleme reel (constate sur le projet Cathodix)

1. **Regressions silencieuses** : Claude modifie un fichier pour corriger un bug, mais sa correction reecrit du code et **perd des features ajoutees precedemment**. Ca compile, les tests passent, mais le comportement a change.

2. **Features qui cassent d'autres features** : modifier le service A change une signature ou un retour, et le controller B qui l'utilise ne fonctionne plus correctement ‚Äî mais pas d'erreur de compilation.

3. **Claude ne connait pas l'historique** : chaque instance Claude Code demarre sans savoir ce qui a ete modifie recemment sur un fichier. Il ne sait pas que `calculateTotal()` a ete enrichie la semaine derniere ‚Äî il peut la reecrire dans une version simplifiee.

4. **Erreurs silencieuses** : `catch {}` vides qui masquent des bugs pendant des semaines.

### L'approche

CodeGuard v0.1 sait **ce qui existe** (carte du projet). La v0.2 doit savoir **ce qui a change** (historique) et **ce qui est casse** (detection de regressions). Le tout injecte automatiquement via les hooks existants ‚Äî Claude n'a pas a "penser" a lire les regles.

## Ameliorations a implementer

### 1. `guard` enrichi ‚Äî historique git + hotspots + couverture tests

**Fichier** : `src/tools/guard.ts`

**Objectif** : avant chaque modification, Claude voit l'historique recent du fichier, les fonctions modifiees recemment, et lesquelles ont des tests.

#### 1a. Historique git par fonction

Ajouter dans `GuardResult` :

```typescript
export interface FileHistory {
  /** Derniers commits qui ont touche ce fichier (max 10) */
  recentCommits: {
    hash: string;
    date: string;
    message: string;
    /** Fonctions dont les lignes sont dans le diff de ce commit */
    functionsChanged: string[];
  }[];
  /** Nombre total de modifications dans les 30 derniers jours */
  modificationCount30d: number;
}
```

**Implementation** :
1. Executer `git log --oneline --since="30 days ago" --follow -- <filePath>` pour obtenir les commits recents
2. Pour chaque commit, executer `git diff <commit>~1 <commit> -- <filePath>` et extraire les numeros de lignes modifiees
3. Croiser les lignes modifiees avec les `functions` du `FileNode` (chaque fonction a un champ `line`) pour determiner quelles fonctions ont ete touchees
4. Limiter a 10 commits max pour ne pas surcharger

**Affichage dans `formatGuardResult`** :

```
### Historique recent
- 5 avr (a1b2c3) feat: ajout calcul TVA ‚Üí calculateTotal(), applyTax()
- 3 avr (d4e5f6) fix: correction arrondi ‚Üí roundDecimal()
‚öÝ Ces fonctions ont ete modifiees recemment. Ne pas perdre ces changements.
```

#### 1b. Hotspot detection

Ajouter dans `GuardResult` :

```typescript
/** true si le fichier a ete modifie >= 5 fois en 7 jours */
isHotspot: boolean;
```

**Implementation** : compter les commits des 7 derniers jours avec `git log --oneline --since="7 days ago" -- <filePath> | wc -l`.

**Affichage** :

```
‚öÝ HOTSPOT ‚Äî modifie 8 fois en 7 jours. Fichier instable, considerer le stabiliser avant d'y toucher.
```

#### 1c. Couverture tests par fonction

Ajouter dans `GuardResult` :

```typescript
export interface FunctionCoverage {
  functionName: string;
  line: number;
  /** Fichier de test qui semble couvrir cette fonction (ou null) */
  testFile: string | null;
}
```

**Implementation** :
1. Pour le fichier `src/services/calculate.service.ts`, chercher les fichiers de test correspondants :
   - `src/services/calculate.service.spec.ts`
   - `test/**/calculate*`
   - `tests/**/calculate*`
2. Si un fichier de test est trouve, lire son contenu et chercher les noms de fonctions du service (grep simple : `describe('calculateTotal')`, `it('should calculateTotal')`, `test('calculateTotal')`)
3. Chaque fonction du service est marquee comme "testee" ou "non testee"

**Precision attendue** : ~70-80%. Ne detecte pas les tests indirects (un test qui appelle un controller qui appelle le service). Mais c'est suffisant pour signaler les fonctions non testees.

**Affichage** :

```
### Couverture tests
- calculateTotal() ‚Üí calculate.service.spec.ts ‚úì
- roundDecimal() ‚Üí calculate.service.spec.ts ‚úì
- applyDiscount() ‚Üí PAS DE TEST ‚úó
‚öÝ Modifier applyDiscount() sans test = aucun filet de securite.
```

### 2. `check` enrichi ‚Äî signatures + code mort + patterns

**Fichier** : `src/tools/check.ts`

#### 2a. Detection changement de signature + appelants casses

**Objectif** : si une fonction change de signature (nombre de params, types), lister tous les fichiers qui l'appellent et qui ne sont pas mis a jour.

Ajouter dans `CheckResult` :

```typescript
export interface SignatureChange {
  functionName: string;
  oldParams: string;    // ex: "(amount: number, rate: number)"
  newParams: string;    // ex: "(amount: number)"
  callers: string[];    // fichiers qui appellent cette fonction
}
```

**Implementation** :
1. Comparer `oldNode.functions` avec `newNode.functions` pour chaque fonction exportee
2. Pour chaque fonction dont les parametres ont change :
   - Chercher tous les fichiers qui importent ce fichier (`graph.getDependents()`)
   - Grep dans ces fichiers pour trouver ceux qui appellent la fonction (`functionName(`)
3. Signaler comme `error` si le nombre de params a diminue ou si un param required a ete ajoute

**Affichage** :

```
### Signatures modifiees
/!\ calculateTotal() : signature changee
    Avant : (amount: number, rate: number)
    Apres : (amount: number)
    Appelants a verifier :
    - src/controllers/order.controller.ts
    - src/services/invoice.service.ts
    ‚öÝ REGRESSION PROBABLE ‚Äî ces fichiers appellent encore avec 2 params
```

#### 2b. Detection code mort apres modification

**Objectif** : apres une modification, detecter les exports qui ne sont plus importes par personne.

**Implementation** :
1. Apres re-indexation du fichier modifie, reconstruire le graphe
2. Pour tous les fichiers du projet, verifier si chaque export a au moins un importeur
3. Les exports sans importeur dans des fichiers non-entry-point = probablement mort
4. Exclure les entry points (index.ts, main.ts, app/, route.ts) et les fichiers de test

**Note** : `health.ts` fait deja la detection d'orphelins au niveau fichier. Ici c'est au niveau **export** ‚Äî plus granulaire.

**Affichage** :

```
### Code mort detecte
- calculateOldTotal() dans calculate.service.ts n'est plus appele par personne ‚Üí supprimer ?
```

#### 2c. Detection d'incoherence de patterns

**Objectif** : si la majorite des fichiers dans un dossier suivent un pattern, signaler quand un fichier s'en ecarte.

**Patterns a detecter** (par analyse statistique, pas par heuristique) :

| Pattern | Comment detecter | Seuil |
|---|---|---|
| Logger dans les catch | Compter les fichiers du dossier qui ont `logger.(error\|warn)` dans un CatchClause | Si > 70% l'ont et ce fichier non ‚Üí signaler |
| Validation zod/class-validator | Compter les imports de `zod` ou `class-validator` dans les fichiers du meme dossier | Si > 70% l'ont et ce fichier non ‚Üí signaler |
| Decorateurs NestJS | Compter les classes avec `@Injectable()`, `@Controller()` | Si le dossier est "services/" et ce fichier n'a pas `@Injectable()` ‚Üí signaler |

**Implementation** :
1. Identifier le dossier parent du fichier modifie
2. Lister tous les fichiers `.ts` du meme dossier (freres)
3. Pour chaque pattern, compter combien de freres l'ont
4. Si > 70% des freres ont le pattern et le fichier modifie non ‚Üí warning

**Precision attendue** : bonne sur les projets bien structures (comme Cathodix qui utilise NestJS avec des patterns constants). Faux positifs possibles sur les petits dossiers (< 3 fichiers) ‚Äî ignorer dans ce cas.

**Affichage** :

```
### Incoherence avec les fichiers du meme dossier
! 8/9 services utilisent logger.error dans les catch ‚Äî celui-ci non
! 7/9 services valident les entrees avec zod ‚Äî celui-ci non
```

### 3. `whatsnew` ‚Äî nouvel outil

**Fichier a creer** : `src/tools/whatsnew.ts`

**Objectif** : au debut d'une session, montrer ce qui a change dans le projet depuis le dernier reindex. La "memoire du projet" pour les nouvelles instances.

**Input** :

```typescript
{
  name: 'whatsnew',
  description: 'Resume des changements dans le projet depuis le dernier reindex. A lancer en debut de session pour comprendre le contexte.',
  inputSchema: {
    type: 'object',
    properties: {
      since: {
        type: 'string',
        description: 'Date ou duree (ex: "3 days ago", "2026-04-05"). Defaut: date du dernier reindex.',
      },
    },
  },
}
```

**Output** :

```typescript
export interface WhatsnewResult {
  since: string;
  filesModified: number;
  filesAdded: number;
  filesDeleted: number;
  /** Fonctions dont la signature a change */
  signatureChanges: { file: string; function: string; oldSig: string; newSig: string }[];
  /** Nouvelles routes API */
  newRoutes: { method: string; path: string; file: string }[];
  /** Routes supprimees */
  removedRoutes: { method: string; path: string }[];
  /** Fichiers les plus modifies (top 5) */
  hotFiles: { file: string; commitCount: number }[];
}
```

**Implementation** :
1. Lire la date du snapshot (`snapshot.json` ‚Üí `indexedAt`)
2. `git log --oneline --since=<date>` pour lister les commits
3. `git diff --name-status <oldest-commit>..HEAD` pour les fichiers ajoutes/modifies/supprimes
4. Comparer snapshot vs index actuel pour les signatures et routes changees (reutiliser la logique de `changelog.ts`)
5. Compter les commits par fichier pour les hotspots

**Affichage** :

```
## Quoi de neuf (depuis le 5 avril)

**12 fichiers modifies** | 3 ajoutes | 1 supprime

### Signatures modifiees
- calculateTotal() dans calculate.service.ts : (amount) ‚Üí (amount, rate)
  ‚öÝ Verifier les 5 appelants

### Nouvelles routes
- POST /api/invoices ‚Äî invoice.controller.ts
- GET /api/invoices/:id ‚Äî invoice.controller.ts

### Fichiers les plus actifs
| Fichier | Modifications |
|---|---|
| calculate.service.ts | 8 commits |
| order.controller.ts | 5 commits |
| invoice.service.ts | 3 commits |
```

## Limites connues

| Limite | Pourquoi | Impact |
|---|---|---|
| Git doit etre initialise | Les commandes git echouent sans .git/ | Verifier avant d'appeler git.ts |
| Historique git = commits, pas sessions Claude | Si Claude fait 3 modifs dans un seul commit, on voit 1 entree | Moins granulaire qu'ideal |
| Couverture tests = heuristique | Tests indirects non detectes | Faux negatifs possibles |
| Patterns = statistique sur le dossier | Petit dossier (< 3 fichiers) = pas fiable | Ignorer les dossiers < 3 fichiers |
| Signatures = comparaison de strings | Pas de verif semantique (ex: type alias) | Faux positifs rares |
| `execSync` = bloquant | Commandes git synchrones | Acceptable pour des commandes rapides |
